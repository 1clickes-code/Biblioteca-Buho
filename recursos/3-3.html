01

La arquitectura: cuatro carpetas que lo hacen funcionar

Antes de lanzar ningún agente, el proyecto tiene que estar en orden. Un agente que arranca sin esta estructura trabaja sin brújula — produce resultados, pero no puede coordinarse con otros agentes ni mantener coherencia entre sesiones.

▶ tu-proyecto/
├── CLAUDE.md ← empieza aquí. Sin esto no hay agente
├── context/ ← quién eres y a quién le hablas
│  ├── perfil.md ← tu voz, tu tono, tu forma de trabajar
│  ├── audiencia.md ← a quién va dirigido el output
│  └── referencias/ ← ejemplos del resultado que quieres
├── output/ ← donde el agente deja lo que produce
│  ├── investigacion/
│  ├── guiones/
│  └── titulos/
└── .claude/skills/ ← instrucciones especializadas por tarea
   └── [nombre-skill]/
      └── skill.md
Por qué el orden importa

Claude Code lee el CLAUDE.md primero, siempre. Luego el agente accede al contexto que necesita. La carpeta output/ es el canal de comunicación entre agentes — si no existe, los agentes no pueden pasarse información. Créala antes de lanzar el primero.

02

El CLAUDE.md: el primer archivo que escribe el arquitecto

Es el documento más importante de cualquier proyecto con agentes. Claude Code lo lee en cada sesión antes de hacer cualquier cosa. Si no existe, el agente empieza sin saber qué proyecto es, qué tiene que producir ni cómo tiene que comportarse.

No necesita ser largo. Necesita ser claro en cuatro cosas:

Ejemplo — CLAUDE.md mínimo y funcional
# Proyecto: [nombre del proyecto]

## Qué es este proyecto
[Una frase. Qué se produce aquí y para quién.]

## Objetivo del agente
[Qué tiene que conseguir. Acotado. Sin ambigüedad.]

## Cómo funciona la estructura
- Contexto: context/perfil.md + context/audiencia.md
- Resultados: dejar siempre en output/[carpeta]/
- Skills disponibles: .claude/skills/[nombre]/skill.md

## Reglas de operación
- Una skill = una tarea. No asignar más de las necesarias.
- No modificar archivos fuera del proyecto.
- Verificar el output antes de marcarlo como entregado.
La prueba del CLAUDE.md

Si alguien que no conoce el proyecto lee tu CLAUDE.md y en 30 segundos sabe qué produce, cómo está estructurado y qué reglas sigue el agente — está bien escrito. Si necesita preguntar, está incompleto.

03

Skills: una por tarea, no más

Una skill es un archivo Markdown con instrucciones especializadas para una tarea concreta. El agente la carga cuando la necesita. Cuantas más skills asignes a un solo agente, peor trabaja — porque tiene que decidir entre instrucciones que compiten entre sí.

La regla que no cambia
Una skill = una tarea.
Dos tareas = dos skills. Dos skills = dos agentes.
Si el agente necesita saber más de tres cosas para hacer su trabajo, son dos agentes, no uno.

La respuesta a "¿y si quiero más de una tarea?" no es una skill más — son dos agentes coordinados. Cada uno especializado. Coordinados vía output/.

Cómo crear una skill
1
Abre Claude Code y ejecuta /plugins
En cualquier entorno (terminal, VS Code, Antigravity). El menú de plugins aparece con las opciones disponibles.
2
Selecciona Skill Creator
El plugin que viene integrado en Claude Code. Te guía en la creación de la skill con preguntas: nombre, objetivo, instrucciones clave.
3
Define una sola cosa que hace la skill
Responde a "¿qué hace?" en una frase. Si necesitas usar "y" para describirla, es dos skills. Skill Creator genera el archivo y lo guarda en la ruta correcta.
4
Verifica la ruta
La skill queda en .claude/skills/[nombre]/skill.md. Si necesitas usarla también en Antigravity, copia a .agents/skills/.
Alternativa manual

Crea el archivo directamente: .claude/skills/[nombre]/skill.md. Escribe el objetivo en la primera línea, las instrucciones en el cuerpo. Skill Creator lo hace más rápido, pero no es obligatorio.

04

Crear el agente con /agents

Con la arquitectura montada y al menos una skill creada, el agente se configura desde Claude Code en cuatro campos. Cada campo tiene su razón — ninguno es decorativo.

/
agents
Campo
Qué define
Ejemplo
Modelo
La potencia y el coste. Sonnet como punto de entrada: equilibra calidad y consumo del plan. Haiku para tareas repetitivas y simples. Opus solo si el resultado necesita razonamiento complejo.
claude-sonnet — punto de entrada recomendado
Color
Diferenciación visual en la terminal cuando varios agentes corren a la vez. No afecta al comportamiento — sí facilita la supervisión.
Amarillo para el agente principal
Scope
A qué tiene acceso. "Proyecto" es la opción correcta para empezar: el agente lee y escribe dentro del directorio del proyecto, nada más. No expandas el scope sin razón concreta.
Proyecto (no global)
Instrucciones
Qué tiene que hacer este agente. Una sola responsabilidad. Cuanto más acotado, mejor funciona. El agente guarda estas instrucciones como fichero Markdown en la carpeta agent/.
Investiga tendencias del tema dado. Lee @context/audiencia.md. Deja el briefing en output/investigacion/.
Lo que hace /agents internamente

Crea un fichero Markdown en la carpeta agent/ con la configuración completa. Puedes editarlo directamente si necesitas ajustar instrucciones sin volver al menú.

05

Ejecutarlo: el @archivo y el error del 80%

El agente está configurado. Para lanzarlo, escribes la instrucción en el chat de Claude Code y referencias los archivos que necesita con el símbolo @. Ese símbolo es la forma en que los agentes se pasan contexto entre ellos.

Sintaxis de ejecución
# Lanzar el agente con contexto específico
Investiga los últimos ángulos sobre automatización de contenido.
Usa la información de @context/audiencia.md como referencia.
Deja el resultado en output/investigacion/briefing.md

# El siguiente agente recoge ese resultado
Genera el guion completo usando el briefing de
@output/investigacion/briefing.md
Sigue las instrucciones de voz en @context/perfil.md

Los agentes no se comunican directamente entre sí — se pasan contexto a través de archivos en output/. Cada agente lee el output del anterior con @ y produce el suyo. Esa es la cadena.

→
output/investigacion/briefing.md
Agente investigador lo produce
→
output/guiones/guion.md
Agente guionista lo lee y produce
→
output/titulos/opciones.md
Agente distribuidor lo lee y produce
El error que comete el 80%
Error más común — y más costoso

Pedir al agente que investigue Y redacte Y genere títulos Y optimice en una sola instrucción. El resultado es mediocre en todo. El agente tiene que distribuir su atención entre tareas que compiten — y ninguna sale bien. Un agente = una especialización. Si necesitas tres cosas, son tres agentes.

El agente que lo hace todo
no hace bien ninguna cosa.
Cómo verificar que funciona
1
Revisa que el archivo de output existe
El agente tiene que haber dejado algo en la carpeta que le indicaste. Si la carpeta está vacía, el agente no completó la tarea — revisa las instrucciones y el scope.
2
Lee el output antes de pasarlo al siguiente agente
Un mal briefing produce un mal guion. Un buen briefing produce un buen guion. El output de cada agente es el input del siguiente — supervisar ese punto es trabajo humano, no opcional.
3
Si algo no funciona, empieza por el CLAUDE.md
El 90% de los fallos del agente tienen origen en instrucciones ambiguas o contexto insuficiente. Antes de cambiar el modelo o la skill, aclara las instrucciones. Un CLAUDE.md más específico resuelve más problemas que un modelo más potente.
