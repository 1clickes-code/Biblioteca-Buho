01

Por qué la palabra "agente" confunde más que aclara

Si buscas "agente de IA" vas a encontrar tres cosas completamente distintas llamadas igual: un chatbot con memoria de conversación, una automatización de N8N que encadena pasos, y un sistema que toma decisiones autónomas sobre cómo resolver un problema.

El término se usa para venderlo todo porque suena bien. El problema es que cuando no sabes exactamente qué es, no puedes decidir cuándo lo necesitas — ni darte cuenta cuando te están vendiendo algo que no lo es.

La definición que funciona

Un agente de IA es un sistema que recibe un objetivo, ejecuta acciones para lograrlo, evalúa el resultado y decide si continuar o entregar — sin necesitar que un humano pulse "siguiente" en cada paso.

Lo que define a un agente no es la tecnología que usa. Es el loop. Si no hay loop, no hay agente.

Y el loop es tan bueno como lo que le das al entrar.

Garbage in
"Haz algo útil" → Agente ? → Output genérico
Itera sin mejorar. Terminas supervisando cada paso.
Contexto rico
goals + CLAUDE.md + refs → Agente → → Output preciso
Entrega en menos bucles. Solo revisas el resultado.
Por eso existe el comando goals en Claude Code

Anthropic lo codificó directamente en el producto: una instrucción dedicada para que el agente sepa qué tiene que conseguir y cómo trabajar antes de empezar el bucle. Si el contexto no importara, no existiría.

02

La cadena que todo agente sigue, sin excepción

Da igual la complejidad del sistema. Todo agente de IA sigue la misma cadena de pasos. Entenderla es entender por qué falla — y dónde.

01 · OBJETIVO
Recibe la instrucción
El agente recibe una tarea con un criterio de éxito claro. Sin criterio claro, el agente no sabe cuándo parar — y sigue iterando hasta consumir el contexto o entregar algo inaceptable.
02 · ACTÚA
Usa las herramientas disponibles
Puede usar skills, buscar información, generar contenido, llamar a MCPs externos. Aquí es donde entran las skills: son las herramientas que el agente puede activar en esta fase.
03 · RESULTADO
Obtiene un output intermedio
El resultado de esta iteración. Puede ser un borrador, un análisis, un archivo generado. No es el output final — es lo que tiene hasta ahora.
04 · VERIFICA
Evalúa contra el objetivo
El agente compara el resultado con el criterio de éxito. ¿Está listo? Entrega. ¿No está listo? Vuelve al paso 02, ajusta y lo intenta de nuevo.
05 · ENTREGA
Sale del loop cuando el resultado es válido
El agente termina cuando el resultado cumple el criterio definido en el paso 01 — no antes. Esa es la diferencia con ejecutar un script o una skill suelta.
Un agente no es magia. Es lógica encadenada que se repite hasta que el trabajo está bien hecho.
03

Skill vs Agente: la diferencia que cambia todo

En el Nivel 2 construiste sistemas con skills. Una skill es un conjunto de instrucciones especializadas que Claude carga cuando las necesita — y eso la hace muy potente para tareas concretas. Pero una skill no es un agente.

La diferencia no es de complejidad. Es de arquitectura:

Skill Agente sin skills Agente con skills
Loop autónomo No Sí Sí
Especialización en tarea Sí No Sí
Decide cuándo actuar No Sí Sí
Verifica su propio resultado No Limitado Sí
Puede usar otras herramientas No Limitado Sí

El agente usa las skills como herramientas dentro de su loop. La skill le dice cómo hacer algo concreto — escribir copy, revisar código, generar títulos. El agente decide cuándo usar esa skill, la ejecuta, evalúa el resultado y decide si repetirla o pasar al siguiente paso.

La forma más fácil de verlo

La skill es el martillo. El agente es el carpintero. El carpintero sabe cuándo coger el martillo, cuántas veces golpear y si el resultado es suficientemente bueno. El martillo solo no construye nada solo.

04

RAG vs Agente con skills: dos arquitecturas distintas

RAG (Retrieval-Augmented Generation) es otra arquitectura que se confunde frecuentemente con agentes. No son lo mismo.

Un sistema RAG hace una sola cosa muy bien: busca información relevante en una base de conocimiento y la incluye en el contexto de la respuesta. Busca → responde → se detiene. No hay loop. No hay verificación. No ejecuta acciones sobre el mundo.

Sistema RAG Agente con skills
Qué hace Recupera información de una base de conocimiento y responde Recibe un objetivo, actúa, verifica y entrega
Loop No Sí
Ejecuta acciones externas No — solo lee Sí — lee, escribe, genera, conecta
Mejor para Preguntas sobre documentos propios Tareas con múltiples pasos y decisiones
Puede incluir al otro No Sí — RAG puede ser una skill del agente

La clave: RAG es una técnica, no una arquitectura completa. Un agente puede usar RAG como una de sus herramientas — busca información en los documentos relevantes y la usa en la siguiente iteración del loop. Pero un sistema RAG solo no puede ser un agente, porque no tiene loop ni capacidad de acción.

Cuándo usar cada uno

¿Quieres preguntarle a tus documentos y obtener una respuesta? RAG. ¿Quieres que el sistema tome una tarea y la complete de principio a fin iterando hasta que esté bien? Agente con skills.

05

El modelo que eliges y la pregunta que tienes que hacerte

No todos los agentes necesitan el mismo modelo. Elegir mal aquí no rompe el agente — lo hace caro, lento, o poco preciso en los momentos que importan.

Haiku
Rápido
Tareas repetitivas, muchas iteraciones, clasificación, extracción de datos. Económico — útil cuando el volumen importa más que la profundidad.
Sonnet
Equilibrado
El punto de entrada recomendado para la mayoría de agentes. Producción de contenido, análisis, sistemas de trabajo. La mejor relación calidad/coste.
Opus
Potente
Razonamiento complejo, decisiones de alta responsabilidad, investigación profunda. Consume el plan más rápido — úsalo donde la calidad no admite compromiso.

Y antes de construir cualquier agente, hay una pregunta que siempre merece una respuesta honesta:

01
¿La tarea requiere múltiples pasos que se verifican entre sí?
Si es una sola acción con un input y un output claro, una skill o un MCP es suficiente. No necesitas un agente.
02
¿El sistema necesita tomar decisiones basadas en resultados intermedios?
Si el siguiente paso siempre es el mismo independientemente del resultado anterior, es una automatización lineal — no un agente.
03
¿El criterio de "listo" es suficientemente claro para que el sistema lo evalúe sin intervención humana?
Si el criterio es subjetivo o requiere juicio humano en cada iteración, el agente no puede cerrar el loop solo — y vas a terminar supervisándolo más de lo que ahorras.
